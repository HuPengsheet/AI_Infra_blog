# AI 推理链路的延迟与 SLO

AI 服务的慢，不一定是模型本身慢。很多时候，真正的问题来自链路中的多个环节叠加：

- 网关排队
- 检索阶段抖动
- 模型调度等待
- GPU 冷启动
- 下游工具调用超时

## 一个可落地的拆分方式

把总时延按阶段拆开：

1. Request Ingress
2. Retrieval / Rerank
3. Prefill
4. Decode
5. Post Process

这样你就可以清楚判断，到底是“模型慢”，还是“模型前后链路慢”。

## 推荐保留的几类指标

- 请求级：QPS、错误率、P50/P95/P99
- token 级：input tokens、output tokens、tokens/s
- 资源级：GPU 利用率、显存占用、队列长度
- 版本级：模型版本、prompt 模板版本、索引版本

## 什么时候算达成 SLO

SLO 不是一句“接口别挂”就够了，更适合写成类似：

> 在最近 7 天内，95% 的在线问答请求在 4 秒内完成，成功率不低于 99.5%。

这种定义的好处是你可以直接把它绑定到报警、发布门禁和回归对比。

## 个人站点里为什么也值得记录这些

因为很多高价值经验不在“最终结论”，而在：

- 指标怎么拆的；
- 监控是怎么埋的；
- 故障时先看哪个面板；
- 发布前做了哪些回归检查。

这些内容非常适合持续写成笔记，后续无论面试、分享还是团队内部复盘都会很好用。
